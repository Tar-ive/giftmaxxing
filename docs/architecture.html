1 · Current planAS DESIGNED

The straightforward pipeline from CLOUD.md v1: ingest → S3 → embedder Lambda → Bedrock → DynamoDB, with a brute-force cosine ranker in the API Lambda. Simple, ~$0 extra, good to ~10k–100k vectors. Review this first.

2 · OptimizedWELL-ARCHITECTED

Hardens the write path and swaps the DIY index for the real AWS vector DB. Changes from #1 are listed below the diagram.

What changed & why (sanity checks + optimizations):

  • Real vector DB → Amazon S3 Vectors replaces brute-force-in-Lambda as the canonical index (confirmed available, lowest cost, scales past 100k). OpenSearch Serverless added only as an optional low-latency hot tier (it has a monthly OCU minimum — keep it out of the critical path until scale demands it).
  • Decouple ingest from embed with SQS + DLQ. Pinterest Trial caps at 1,000 req/day — a queue smooths bursts, enables retries/backoff, and isolates failures (DLQ + alarm).
  • Step Functions orchestration instead of Lambda-chaining the multi-step ingest→dedup→embed→upsert flow (visibility, retries, idempotency).
  • Perceptual-hash dedup before embedding — skip near-duplicate pins to cut cost + feed noise.
  • Bedrock BATCH for backfills (50% cheaper) vs on-demand for incremental real-time embeds.
  • EventBridge Scheduler for periodic re-sync of connected boards (events trigger work, not cron-in-Lambda).
  • Secrets Manager for Pinterest/affiliate keys (not .env in prod).
  • Idempotency by content hash so re-runs never re-embed the same image; cache (DAX) hot recommendations; CloudFront for image delivery.
  • Observability: CloudWatch metrics/alarms (esp. DLQ depth) + X-Ray traces + an AWS Budgets alarm as a cost guardrail.
  • Visual search reuses the same vector DB — image query → same Titan model → kNN → affiliate enrich.
  • Consider 256/384-d embeddings (Titan supports it) to cut vector storage/query cost if recall stays acceptable.
